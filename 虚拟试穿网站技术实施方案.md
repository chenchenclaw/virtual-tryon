# AI 虚拟试穿网站 — 技术实施方案

## 一、项目概述

### 1.1 产品愿景

打造一个基于 AI 的在线虚拟试穿平台，用户上传个人照片和身材数据后，可自由搭配上衣、裤装、鞋履等单品，由 AI 实时生成逼真的全身穿搭效果图。平台旨在解决线上购衣"看不见上身效果"的核心痛点，降低退货率，提升购物体验。

### 1.2 核心价值

- **所见即所得**：用户无需实物即可预览穿搭效果
- **自由搭配**：支持多品类、多风格混搭组合
- **个性化推荐**：基于身材数据和偏好智能推荐适配单品
- **社交分享**：穿搭方案可一键分享至社交平台

## 四、AI 核心技术方案

### 4.1 方案概述

本方案采用 **GPT Image + 智能体（Agent）** 架构实现虚拟试穿，核心思路是：

> 将用户照片、身材数据和服装信息组装为结构化 Prompt，通过智能体调度 OpenAI GPT Image（图片编辑/生成）API，直接输出试穿效果图。

与传统方案（自建人体解析 → 姿态估计 → 服装分割 → 试穿模型流水线）相比，本方案：
- **无需部署任何 AI 模型**，不依赖 GPU 集群
- **无需训练定制模型**，利用 GPT Image 的通用图像理解与生成能力
- **开发周期短**，核心工作在于 Prompt 工程和质量调优
- **迭代成本低**，调整 Prompt 即可优化效果，无需重新训练

### 4.3 Prompt 工程设计（核心）

Prompt 质量直接决定试穿效果，采用 **分层模板 + 动态组装** 策略。

#### 4.3.1 Prompt 结构分层

```
┌─────────────────────────────────────────────────┐
│  System Prompt（固定指令层）                       │
│  "你是一个专业的虚拟试穿 AI 助手。                  │
│   请根据用户提供的照片和数据，生成逼真的穿搭效果图。   │
│   保持用户面部特征、发型、体型比例不变。..."          │
├─────────────────────────────────────────────────┤
│  User Prompt（动态内容层）                         │
│                                                   │
│  [人物描述] 用户全身正面照 + 身材数据文本描述         │
│  [服装描述] 各单品照片 + 属性标签文本               │
│  [搭配指令] 指定哪些单品穿在哪些部位                 │
│  [场景/姿势] 背景场景 + 模特姿势                    │
│  [质量要求] 分辨率、风格、细节要求                   │
├─────────────────────────────────────────────────┤
│  Reference Images（参考图片）                      │
│  - 用户全身照片（必传）                             │
│  - 上衣照片（可选，多件）                           │
│  - 裤装照片（可选）                                │
│  - 鞋履照片（可选）                                │
└─────────────────────────────────────────────────┘
```

#### 4.3.2 Prompt 模板示例

**System Prompt**：

```
你是一个专业的虚拟试穿 AI。你的任务是根据用户的照片和服装信息，
生成用户穿着指定服装的逼真效果图。

核心规则：
1. 必须保持用户的脸部特征、发型、肤色完全一致
2. 必须根据提供的身材数据调整体型比例，确保合身感
3. 服装的颜色、图案、材质必须忠实还原参考图
4. 光影和环境要自然协调
5. 输出为高质量摄影风格图片
```

**User Prompt 模板**：

```
## 用户信息
- 身高：{height}cm，体重：{weight}kg
- 肩宽：{shoulder}cm，胸围：{chest}cm，腰围：{waist}cm，臀围：{hip}cm
- 体型特征：{body_type_description}
- 风格偏好：{style_preference}

## 穿搭要求
请让图中人物穿上以下服装：
- 上衣：{garment_top_description}（见附件图片1）
  · 推荐尺码：{top_size}（对应胸围 {top_chest}cm，衣长 {top_length}cm）
  · 版型：{top_fit}（修身/标准/宽松）
  · 合身预期：{top_fit_description}
- 裤装：{garment_bottom_description}（见附件图片2）
  · 推荐尺码：{bottom_size}（对应腰围 {bottom_waist}cm，裤长 {bottom_length}cm）
  · 版型：{bottom_fit}（修身/直筒/宽松/阔腿）
  · 合身预期：{bottom_fit_description}
- 鞋履：{garment_shoes_description}（见附件图片3）
  · 推荐尺码：{shoes_size}（对应脚长 {shoes_length}cm）

## 立体感与合身度要求
- 服装需根据版型和尺码数据呈现真实的合身效果
- {fit}版型的服装在用户身上应呈现{fit_visual_description}
- 注意服装与身体之间的间隙：修身款贴合身体曲线，宽松款有自然垂坠感
- 服装的褶皱、垂坠、拉伸等细节需符合实际穿着状态

## 场景与姿势
- 背景场景：{scene_description}
- 模特姿势：{pose_description}
- 拍摄角度：{camera_angle}

## 质量要求
- 分辨率：{resolution}
- 风格：{photography_style}
- 特别注意：{special_instructions}
```

```typescript
const MAX_RETRIES = 2;
async function tryonWithRetry(request: TryonRequest): Promise<TryonResult> {
  for (let i = 0; i <= MAX_RETRIES; i++) {
    const result = await generateTryon(request);
    const check = await qualityCheck(request.userPhoto, result.image);

    if (check.pass || i === MAX_RETRIES) {
      return { ...result, qualityScore: check.score };
    }

    // 根据问题调整 Prompt 后重试
    request = refinePrompt(request, check.issues);
  }
```

### 4.5 身材数据的应用

2. **合身感控制**：通过 "修身"/"宽松"/"标准版型" 等描述控制服装贴合度

#### 4.6.2 尺码推荐算法

根据用户身材数据和服装尺码表，自动推荐最佳尺码：

```typescript
function recommendSize(user: BodyProfile, sizeChart: SizeChart): SizeRecommendation {
  const scores = sizeChart.sizes.map(size => {
    let matchPoints = 0;
    let totalPoints = 0;

    // 根据品类匹配关键维度
    if (size.measurements.chest) {
      totalPoints += 3;  // 胸围权重最高
      const diff = Math.abs(user.chest_circ - size.measurements.chest);
      matchPoints += Math.max(0, 3 - diff / 2);  // 每差2cm扣1分
    }
    if (size.measurements.shoulder) {
      totalPoints += 2;
      const diff = Math.abs(user.shoulder_width - size.measurements.shoulder);
      matchPoints += Math.max(0, 2 - diff / 1.5);
    }
    if (size.measurements.waist_circ) {
      totalPoints += 3;
      const diff = Math.abs(user.waist_circ - size.measurements.waist_circ);
      matchPoints += Math.max(0, 3 - diff / 2);
    }
    if (size.measurements.hip_circ) {
      totalPoints += 2;
      const diff = Math.abs(user.hip_circ - size.measurements.hip_circ);
      matchPoints += Math.max(0, 2 - diff / 2);
    }

    return {
      size_label: size.size_label,
      score: matchPoints / totalPoints,
      fit_analysis: generateFitAnalysis(user, size.measurements),
    };
  });

  // 按匹配度排序，返回最佳尺码
  scores.sort((a, b) => b.score - a.score);
  return {
    recommended: scores[0],
    alternatives: scores.slice(1, 3),
    explanation: generateSizeExplanation(user, scores[0]),
  };
}
```

### 4.7 版型与姿态估计（二期，详见独立文档）

一期方案中，版型信息（修身/宽松等）通过文本描述注入 Prompt，已有一定效果。但客户指出：不同体态（含胸、溜肩、O 型腿等）对版型呈现影响很大，仅靠围度数据和文字描述不够精确。

二期将引入 **姿态估计**，从用户照片中自动提取体态特征，结合服装版型信息，生成更立体、更真实的试穿效果。核心技术路径：

```
用户照片 → 姿态估计(MediaPipe) → 体态特征提取 → 版型×体态矩阵匹配 → Prompt 增强
```

关键技术点：
- 姿态估计采用 MediaPipe Pose（浏览器端运行，无需后端 GPU）
- 建立 "版型 × 体态" 矩阵，针对不同组合生成差异化的合身描述
- 在生成效果图上叠加合身度可视化标注

> 详细技术方案、算法设计和开发计划见：《虚拟试穿二期 — 版型与姿态估计技术方案》

#### 4.9.2 多图输入策略

当需要传入多张服装参考图时，有以下策略：

| 策略 | 实现方式 | 适用场景 |
|------|---------|---------|
| **拼图法** | 将多张服装图拼接为一张合图作为输入 | 单次 API 调用，最简单 |
| **分步法** | 先试穿上衣，结果再作为输入加裤装 | 效果最好，但多次调用成本高 |
| **描述法** | 仅用文字描述服装，不传服装图 | 服装图质量差时的降级方案 |
| **混合法** | 传关键单品图 + 文字补充其他单品 | **推荐方案**，平衡效果与成本 |

**推荐：混合分步法**

```
第一步：用户照片 + 上衣图片 → 生成穿上衣的效果图
第二步：上衣效果图 + 裤装图片 → 生成完整搭配效果图
第三步（可选）：完整效果图 + 鞋履图片 → 生成带鞋的最终效果图
```

每步传递上一步的结果作为新输入，逐步叠加服装层次。

### 4.10 效果优化策略

#### 4.8.1 Prompt 调优

- **负面指令**：在 Prompt 中加入 "不要改变面部特征"、"不要改变发型"、"不要改变体型" 等约束
- **参考风格**：附加 "参考时尚杂志摄影风格"、"参考电商产品图风格" 等风格锚定
- **细节强调**：对特殊服装细节（刺绣、印花、蕾丝边等）单独描述
- **A/B 测试**：同一搭配用不同 Prompt 表述生成，对比选优

#### 4.8.2 输入图片优化

- **用户照片标准化**：引导用户拍摄标准正面全身照（纯色背景、自然光线、双手自然下垂）
- **服装图片预处理**：自动裁剪、去背景、增强对比度
- **分辨率适配**：输入图片统一缩放到 API 最优分辨率（1024x1536）

#### 4.8.3 结果后处理

- **超分辨率**：如需更高分辨率，可对 API 输出做 2x 超分
- **色彩校正**：确保服装颜色与实物一致（通过颜色参考锚点）
- **水印与版权**：在最终输出图上添加平台水印

### 7.2 实时状态推送方案

采用 **WebSocket + SSE 双通道**：

- **WebSocket**：实时推送任务进度（排队中 → 处理中 → 生成中 → 完成）
- **SSE (Server-Sent Events)**：作为 WebSocket 的降级方案
- **轮询**：作为最终降级方案，每 2 秒查询一次

```typescript
// 前端 WebSocket 监听示例
const ws = new WebSocket(`wss://api.example.com/ws/tryon/${taskId}`);

ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    switch (data.stage) {
        case 'queued':
            updateProgress(10, '排队等待中...');
            break;
        case 'preparing':
            updateProgress(25, '正在准备素材和 Prompt...');
            break;
        case 'generating':
            updateProgress(50, 'AI 正在生成试穿效果...');
            break;
        case 'quality_check':
            updateProgress(80, '正在检查生成质量...');
            break;
        case 'completed':
            updateProgress(100, '生成完成');
            displayResult(data.result_urls);
            break;
        case 'failed':
            showError(data.error_message);
            break;
    }
};
```

## 八、性能与成本优化

### 8.1 性能目标

| 指标 | 目标值 |
|------|--------|
| 单次试穿生成时间 | < 20 秒（单件）/ < 45 秒（多件分步） |
| 并发处理能力 | 30+ 同时在线用户（受 API 并发限制） |
| 图片上传速度 | < 3 秒（5MB 以内） |
| 首屏加载时间 | < 2 秒 |
| API 响应时间（非 AI） | < 200ms |

### 8.2 调用优化策略

1. **请求队列**：BullMQ 控制并发，避免超出 OpenAI API 速率限制
2. **Prompt 缓存**：相同身材数据 + 相同服装的体型描述和服装描述缓存到 Redis，避免重复调用 GPT-4o 生成描述
3. **结果缓存**：同一用户 + 同一搭配 + 同一场景的生成结果缓存，有效期 24 小时
4. **图片预处理**：上传时即完成压缩和标准化，不占用生成时的处理时间
5. **分级生成**：用户可选 "快速预览"（低质量参数，速度快）和 "高清生成"（高质量参数）
6. **批量用户错峰**：非高峰时段预热缓存，高峰时段优先处理付费用户

### 8.3 API 调用量估算

| 操作 | 单次 API 调用 | 说明 |
|------|-------------|------|
| 服装属性识别 | 1 次 GPT-4o Vision | 上传时自动触发，结果缓存 |
| 体型描述生成 | 1 次 GPT-4o | 录入身材数据时触发，结果缓存 |
| 单件试穿 | 1 次 GPT Image | 核心操作 |
| 多件搭配（3件） | 2-3 次 GPT Image | 分步叠加 |
| 尺码试穿对比 | 2 次 GPT Image | 推荐尺码 + 备选尺码各一次 |
| 质量自检 | 1 次 GPT-4o Vision | 可选，默认开启 |

**典型用户单次使用流程**：约 2-4 次 API 调用（含试穿 + 质量检查）

### 8.4 成本估算（月度）

以 OpenAI GPT Image 定价为基准（生成 1024x1536 图片约 $0.04-0.12/张，视质量参数而定）：

| 项目 | 小规模（1000 DAU） | 中规模（10000 DAU） |
|------|-------------------|-------------------|
| 假设日均试穿次数 | 3,000 次/天 | 30,000 次/天 |
| GPT Image API | ¥6,000 - 18,000 | ¥60,000 - 180,000 |
| GPT-4o API（辅助） | ¥500 - 1,000 | ¥3,000 - 8,000 |
| 云服务器（Web/API） | ¥1,000 - 2,000 | ¥2,000 - 5,000 |
| 对象存储 + CDN | ¥500 - 1,000 | ¥2,000 - 5,000 |
| 数据库 + Redis | ¥500 - 1,000 | ¥1,000 - 3,000 |
| **合计** | **¥8,500 - 23,000** | **¥68,000 - 201,000** |

**与自建模型方案对比**：
- 自建方案：固定 GPU 成本高（¥20,000-50,000/月起），但边际成本低
- API 方案：无固定成本，按用量付费，初期成本低，规模化后成本线性增长
- **交叉点约在 5000-8000 DAU**，超过此规模可考虑混合方案（高频场景自建模型 + 长尾场景 API）

### 8.5 成本控制措施

- **免费额度限制**：注册用户每日免费 3 次试穿
- **付费增值**：高清、多角度、无水印等高级功能需付费
- **API 调用监控**：实时监控调用量和成本，设置日/月上限告警
- **Prompt 精简**：优化 Prompt 长度，减少 token 消耗
- **缓存复用**：鼓励用户保存搭配方案，避免重复生成

## 九、安全与隐私

### 9.1 数据安全

- **照片加密存储**：用户照片 AES-256 加密后存储
- **传输加密**：全站 HTTPS，API 请求签名
- **数据隔离**：用户数据逻辑隔离，删除即不可恢复
- **访问控制**：RBAC 权限模型，API Key + JWT 双重鉴权

### 9.2 隐私合规

- **GDPR/个人信息保护法**：用户有权随时删除所有数据
- **照片使用授权**：明确告知照片仅用于试穿生成，不做其他用途
- **第三方 API 数据流声明**：用户照片会传输至 OpenAI API 进行图像生成，需在隐私政策中明确告知并取得用户同意
- **数据最小化**：仅将必要的图片和 Prompt 发送给 API，不传输用户身份信息
- **AI 训练声明**：明确告知用户 OpenAI 可能将输入数据用于模型改进（除非用户通过 API 的 opt-out 机制退出）
- **年龄限制**：未满 16 岁需监护人同意

### 9.3 内容安全

- **图片审核**：接入内容安全 API，过滤不当图片
- **生成审核**：AI 生成结果过安全审核后再展示
- **举报机制**：社区内容支持举报

## 十一、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 生成效果不稳定 | 同一输入多次生成结果差异大 | Prompt 精细化约束 + 质量自检 + 多次生成择优 |
| 面部特征偏移 | 用户长相不像本人 | Prompt 中强化面部保持指令 + 人脸相似度校验 |
| 服装还原度不足 | 颜色/图案/材质与实物不符 | 服装属性精细描述 + 颜色锚点 + 分步生成策略 |
| 尺码推荐不准 | 推荐尺码不合适，用户信任度下降 | 多维度加权匹配 + 用户反馈修正 + 显示匹配度分数供参考 |
| 尺码表录入门槛高 | 用户不愿手动录入详细尺码数据 | 内置品类模板快速填充 + 品牌尺码库自动匹配（二期） |
| 尺码对比效果差异不明显 | 用户看不出 M 和 L 的区别 | 调整 Prompt 强化尺码差异描述 + 辅以合身度文字对比 |
| API 成本失控 | 用户量增长后 API 费用线性增长 | 免费额度限制 + 付费增值 + 结果缓存 + 成本监控告警 |
| API 服务不可用 | OpenAI 服务故障导致全站不可用 | 降级方案（排队等待）+ 备选 API（如 Anthropic Claude Image） |
| 生成速度偏慢 | 多件搭配分步生成耗时长 | 快速预览模式 + 进度实时推送 + 用户预期管理 |
| 隐私合规风险 | 用户照片传至第三方 API | 隐私政策明确告知 + 用户授权 + 数据最小化 |
| 姿势/场景受限 | GPT Image 对复杂姿势控制力有限 | 预设姿势模板 + 渐进式开放 + 用户反馈驱动优化 |

## 十二、扩展方向

1. **混合 AI 方案**：当用户规模增长到一定程度，高频场景（如单件上衣试穿）切换为自建专用模型（如 IDM-VTON），长尾场景（多件搭配、场景切换）继续用 GPT Image API，实现成本最优
2. **视频试穿**：接入视频生成 API（如 Sora/Kling），从静态图片扩展到短视频试穿动画
3. **AI 穿搭顾问**：基于 GPT-4o 构建对话式穿搭助手，用户通过聊天描述场景需求，AI 推荐搭配并直接生成效果图
4. **电商导购**：接入电商平台，试穿满意后一键购买，实现 "试穿即转化" 闭环
5. **品牌定制**：为服装品牌提供 API/SDK，品牌官网可直接集成虚拟试穿能力
6. **多模型路由**：根据场景自动选择最优 API（GPT Image / Flux / Midjourney），实现效果和成本的动态平衡
